GitSense Contexts: Snippets for Code Changes

By Terrence Chen on June 12, 2016

Like snippets and gists, which provides an easy way to share segments of code; GitSense contexts, provides an easy way to share parts of Git's history.  With GitSense contexts, you can easily control what parts of Git's history you would like the end user to see, analyze and search.

For example, if you wanted to show others, what is changing in your 2.0 release, you can create a GitSense context, that only includes relevant branches and file changes for that release.  And by having others view Git's history through that context, you can ensure the changes that they see, will always be relevant to the 2.0 release.

In a nutshell, GitSense contexts, makes it possible for you to put code changes into context for others.  And in this blog post, we'll highlight some of the ways you can use GitSense contexts, to build software better, together.

Support for Bitbucket, GitHub and GitLab

Need to put repos that are hosted on Bitbucket, GitHub and GitLab into context?  No problem.  You can easily combine repos from Bitbucket Cloud/Server, GitHub/GitHub Enterprise and GitLab Cloud/Server.  And in the near future, you'll be able to include repos from Visual Studio Team Services as well.

Context ideas

Here are some of the ways you can use GitSense contexts, to help you and others, to work smarter and faster:

  • Create an active releases context.  With this context, you'll be able to quickly tell what's in a release and what's not.

  • See what your teammates are working on, with a teams context.  With this context, you won't have to sift through unnecessary noise, to see what others are working on.

    And if you are only interested in seeing changes to certains files/directories, include some path and language filters as well.

  • Create an issues/tasks context.  With this context, you'll be able to map issues/tasks to changes, which should come in handy for daily scrum meetings.
