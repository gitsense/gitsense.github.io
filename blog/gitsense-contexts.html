GitSense Contexts: Custom Windows into Git's History

By Terrence Chen on June 12, 2016

Like snippets and gists, which provides an easy way to share segments of code; GitSense contexts, provides an easy way to share selected parts of Git's history.  With GitSense contexts, you'll be able to create windows into Git's history, that only shows what you want others to see.

For example, if you wanted to show others, what is changing in your 2.0 release, you can create a GitSense context, that only includes relevant branches and file changes for that release.  And by having others view Git's history through that context window, you can ensure the changes that they see, will always be relevant to the 2.0 release.

In a nutshell, GitSense contexts, makes it possible for you to put code changes into context for others.  And in this blog post, we'll highlight some of the ways we can use GitSense contexts, to build software better, together.

Bitbucket, GitHub and GitLab Support

Need to put repos that are hosted on Bitbucket, GitHub and GitLab into context?  No problem.  You can easily combine repos from Bitbucket Cloud/Server, GitHub/GitHub Enterprise and GitLab Cloud/Server.  And in the near future, you'll be able to include repos from Visual Studio Team Services as well.

Context Use Cases

Here are some of the ways you can work smarter with GitSense contexts.

  • Create contexts for active releases.  With these contexts, you'll be able to quickly tell what's in a release and what's not.

  • Create a teams context, to see what your teammates are working on.  With this context, you won't have to sift through unnecessary noise from others.

  • Create contexts to track issues/tasks.  With these context, you'll be able to map issues/tasks to code changes, which should come in handy for daily scrum meetings.
